RoomFind.ie UX Review

Research Based Suggestion for Form Creation in RoomFind.ie

A research-backed review of the current new-listing experience, focused on reducing drop-off, improving completion rates, and making the listing flow feel lighter and more intuitive for hosts.

Prepared by: Arinze
Document type: Client-facing research memo
Scope: New listing flow at app/(dashboard)/listings/new

Executive Summary

I reviewed the current RoomFind.ie listing creation flow against established UX guidance from GOV.UK, the CMS Design System, Nielsen Norman Group, Baymard Institute, and Google’s vacation-rental listing guidance. Overall, the flow already has a solid structure: it is broken into steps, clearly labels many fields as required vs optional, and includes helpful touches such as media previews and image optimization.

However, the main risk is not visual quality. The main risk is perceived effort. The current experience can feel longer and more mandatory than it really is. That creates hesitation, especially early in the flow, and can lead to abandonment before users reach the publishing step. At the same time, some fields that are worth keeping, such as the lister's role, need stronger downstream use so they feel justified to the user.

What is working Step-based flow, clear field labelling, strong media experience.
Main friction point Too much perceived effort too early, especially in Basics and the overall 8-step flow.
Highest-priority action Keep only justified fields, make validation more specific, and ensure important inputs are visibly used.

Research Principles Used

Principle What the research says Why it matters here
Ask only what is necessary GOV.UK recommends asking only for information you need and avoiding redundant questions. Every unnecessary field increases perceived effort and can create avoidable drop-off.
Reduce recall NN/g recommends recognition over recall, meaning interfaces should help users with cues rather than forcing them to invent answers from scratch. Early freeform fields like title and description are cognitively heavy if introduced too soon.
Use clear validation CMS and GOV.UK recommend error summaries and field-level feedback instead of vague generic failures. Broad toast errors make users search for what went wrong, which adds friction.
Minimize field burden Baymard consistently finds that reducing field burden improves form usability and completion. An 8-step flow needs to feel lighter than it currently feels, even if many fields are optional.
Support rich listing content Google’s vacation-rental guidance emphasizes rich listing attributes and strong image coverage for competitive performance. RoomFind should not oversimplify core listing quality inputs like photos, price, and location.

Review of the Current Flow

The current listing flow is defined in components/listings/CreateListingForm.jsx and consists of 8 steps: Basics, Property, Location, Media, Financials, Amenities, Preferences, and Availability.

Observation: The actual publishing requirements are lighter than the flow feels. In practice, only some steps are blocking. However, from a user’s point of view, the structure still communicates “long, high-effort form.”

What the current flow does well

  • It breaks the task into stages instead of placing everything on one long page.
  • It distinguishes many fields as required versus optional, which is good practice.
  • It includes useful visual components such as media previews and a stepper.
  • It uses strong defaults for several fields, which reduces effort in later steps.

Where the flow creates unnecessary friction

  • It asks for effort-heavy inputs like title and description very early.
  • Validation is mostly generic and not field-specific.
  • The flow includes some optional steps that still feel central rather than secondary.
  • There is no final review or “check your listing” step before submission.
  • Some fields, such as lister role, appear important conceptually but are not surfaced clearly enough after submission.

Research-Based Recommendations

1. Keep the “role” question, but make its value visible and meaningful

In the current Basics step, the form asks “Who is listing this property?” and blocks progress if the field is empty. Based on product intent, this field is meaningful and should remain because it provides useful context about who is advertising the property.

The issue is not that the field exists. The issue is that users need to see why it matters. At the moment, that value is not clearly surfaced in the listing details experience. For example, the listing details page currently shows host information, but not the lister's stated role.

Recommendation: Keep the role field, but ensure it is persisted and displayed in relevant places such as the listing details page so users see that the question has a purpose.

2. Keep core listing-quality fields, but explain their value more clearly

I would not recommend removing core fields such as photos, address/location, price, deposit, or availability date. These are foundational listing inputs, and Google’s listing guidance supports collecting strong listing attributes and rich media.

What should change is the framing. Users need to understand why these fields matter, how they help their listing perform, and in the case of location, how their information will be used.

Recommendation: Keep these fields, but improve the surrounding explanation and reassurance.

3. Replace generic step-level toasts with inline field errors and a step summary

The current form uses broad error toasts such as “Please fill in all basic details.” This is one of the clearest gaps compared to CMS and GOV.UK guidance, both of which recommend error summaries and field-level error messages.

Broad messages force users to scan the entire screen to find the issue. That increases friction and makes the experience feel less forgiving than it needs to be.

4. Reduce early writing burden by supporting recognition over recall

NN/g’s recognition-over-recall principle is highly relevant here. Asking users to write a title and description early in the flow creates a heavy cognitive task before they have built momentum.

This does not mean those fields should be removed. It means the product should help users complete them more easily through examples, templates, prefilled suggestions, or by moving them later once the factual details are already entered.

5. Reposition advanced optional steps so they feel secondary

The Preferences step is valuable for matching quality, but it is dense and fully optional. In its current position, it still feels like part of the mandatory publishing workload.

Baymard’s field-burden findings suggest that optional complexity still affects the user’s sense of effort, especially when it is visually prominent.

Recommendation: Keep these fields, but frame them more clearly as enhancements rather than prerequisites.

6. Add a final review step before publish

A review step is strongly aligned with GOV.UK’s “Check answers” pattern. It gives users confidence, reduces submission anxiety, and provides one final opportunity to catch missing or weak information.

This is especially useful in a listing flow, where users may want to feel that the final result looks polished before going live.

7. Increase motivation around media without increasing the hard requirement

The current experience requires at least 1 photo, which is a reasonable low-friction minimum. However, Google’s guidance recommends significantly more images for competitive listing performance.

Rather than raising the technical barrier immediately, I would recommend better messaging, for example: “You can publish with 1 photo, but listings with 5 to 8+ photos perform better.”

What I Recommend Removing

Remove: unnecessary friction, not meaningful context. At this stage, I would not recommend removing the role question if the business intends to show who is listing the property.

What I Recommend Keeping

Keep: media requirement, address/location, monthly rent, deposit, and availability date. I would also keep the lister-role question if it is surfaced meaningfully in the listing experience. These are core trust, discoverability, and listing-quality inputs.

Prioritized Next Steps

  1. Keep the role field, but make it visible in the listing experience so it feels justified.
  2. Replace generic toast validation with inline field-level errors and an error summary.
  3. Reduce early writing burden by supporting title/description with prompts or moving them later.
  4. Add a final review step before publish.
  5. Reframe optional advanced sections so they feel additive, not required.
  6. Strengthen media guidance to encourage more complete listings.
